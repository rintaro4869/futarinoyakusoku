# Pairlog セキュリティ確認メモ

このメモは、非エンジニアでも「何を守っているのか」が分かるようにまとめたものです。

## 基本防御（実装済み）

### 1. 知らないサイトから API を勝手に呼びづらくする
- 守っているもの: ログイン後の API を、関係ない外部サイトから雑に叩かれるリスク
- やったこと: Pairlog の正しいURLから来たブラウザ通信だけを許可するようにした
- たとえると: 「関係者以外は裏口から入りにくくする」イメージ

### 2. 弱い秘密鍵で本番が動かないようにする
- 守っているもの: ログイン用トークンの安全性
- やったこと: 本番URLなのに `JWT_SECRET` が短い・未設定のままなら起動時に止まるようにした
- たとえると: 「合鍵が簡単すぎるなら、その鍵では営業開始できない」ようにした

### 3. ログイン試行の連打を少し止める
- 守っているもの: パスワード総当たりや荒らし的な試行
- やったこと: 同じ接続元から短時間にログインや新規登録を試しすぎたら、一時的に待ってもらうようにした
- 制限: ログイン10回/15分、登録5回/15分、リセット要求3回/15分、リセット実行5回/15分
- たとえると: 「短時間で何回もガチャガチャされたら、少しだけ入口を止める」

### 4. API をキャッシュさせない
- 守っているもの: 個人情報やログイン後データが端末や途中経路に残りすぎること
- やったこと: API レスポンスに `no-store` をつけ、ブラウザや中継に保存させにくくした

### 5. パスワード確認の崩れ方を減らす
- 守っているもの: パスワード照合の安全性と、壊れたデータでの例外
- やったこと: PBKDF2-SHA256（10万回）+ ランダム塩 + タイミング安全比較

### 6. トークン（ログイン鍵）の安全な保存
- 守っているもの: 他のアプリからトークンを盗み読みされるリスク
- やったこと: iOS キーチェーン / Android 暗号化領域（expo-secure-store）に保存。平文保存なし

### 7. JWT 有効期限の短縮 + 自動更新
- 守っているもの: トークン漏洩時の被害範囲
- やったこと: 有効期限を 365日 → 7日 に短縮。期限が近づいたら自動で新しいトークンに更新する仕組みを追加
- たとえると: 「1年使える通行証」から「1週間ごとに更新が必要な通行証」に変えた

### 8. Privacy Manifest（Apple 必須）
- 守っているもの: App Store 審査通過とプライバシー宣言の正確性
- やったこと: アプリが使用するシステム API（UserDefaults、起動時刻、ファイル日時）を Apple が求める形式で宣言
- たとえると: 「うちのアプリはこの情報にアクセスしますよ」という申告書

### 9. データ削除の完全化
- 守っているもの: ユーザーの「忘れられる権利」
- やったこと: アカウント削除時に、ポイント・修復・安全アクション・分析・招待コード・メンバーシップをすべて削除。メールとパスワードも消去
- 以前: ユーザーテーブルに削除マークだけ → 関連データが全部残っていた

### 10. パスワードリセット
- 守っているもの: パスワードを忘れたユーザーがアカウントを取り戻せること
- やったこと: メールアドレスで6桁コードを発行 → コード＋新パスワードで再設定できる仕組みを追加
- 注意: メール送信サービスは未連携。連携後に実際にコードが届くようになる

### 11. 緊急停止（メンテナンスモード）
- 守っているもの: 事故・攻撃時にサービスを安全に止められること
- やったこと: 環境変数 `MAINTENANCE_MODE=true` を設定すると、全API が「メンテナンス中」を返す
- たとえると: 「シャッターを下ろす」ボタン

## まだ残る課題

### 近い将来やる価値が高いもの
- メール送信サービスの連携（パスワードリセットのコードを実際にメールで届ける）
- メール認証（登録時にメールアドレスの所有確認）
- Cloudflare 側のレート制限（サーバー再起動に影響されない恒久的な制限）
- アカウントロック（一定回数以上の失敗で一時凍結）

### あると良いもの
- 証明書ピンニング（通信の偽装をさらに難しくする）
- 監査ログの整理
- パスワードの複雑さ要件（数字・記号を1つ以上など）

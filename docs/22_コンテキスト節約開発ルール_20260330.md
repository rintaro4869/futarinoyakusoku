# Pairlog コンテキスト節約開発ルール

## なぜ Codex でも圧縮が起きるか

Codex でも Claude でも、原因は基本的に同じです。
コードが長いこと自体より、1ターンで抱える総コンテキスト量が大きいと圧縮が起きます。

主な要因:

- 長い会話履歴を引きずる
- 大きいコードを何本も同時に読む
- QAチェックリスト全文を毎回貼る
- スクショを大量に貼る
- ログやテスト出力を長く残す
- 1ターンで複数テーマを同時に扱う

つまり、問題は「リポジトリが大きい」よりも「1ターンが広すぎる」ことです。

## 基本方針

1ターン = 1テーマ を徹底する。

悪い例:

- 約束作成
- カレンダー
- 認証
- ごほうび
- API

を同時に見る

良い例:

- 今回は `create-rule.tsx` だけ
- 今回はカレンダーのドットと下余白だけ
- 今回は forgot-password だけ

## 役割分担

### Codex

- 実装
- ローカル検証
- 型チェック
- 単体テスト
- handoff更新

### Claude

- 狭い scoped QA
- build 前の確認
- build / submit

注意:

- Claude は QA で止める
- build を使うかはユーザーが最後に決める

## 推奨の作業単位

### 画面単位

- ホーム
- 約束一覧
- 約束作成 / 編集
- カレンダー
- ごほうび
- 設定
- 認証 / パスワードリセット

### 変更単位

- レイアウトだけ
- データ保存だけ
- 通知だけ
- 余白だけ
- 1つのバグだけ

## Pairlog 向けの最適な進め方

### レベル1: ローカル修正

Codex で 1テーマだけ直す。

例:

- 約束作成画面の schedule セクションだけ
- カレンダーの下余白だけ
- ごほうび複数時の加算ロジックだけ

この時のルール:

- 触るファイルは原則 1〜3本
- 目的は 1つだけ
- 直したら `tsc` と関連テストだけ回す

### レベル2: scoped QA

Claude に投げる時は、差分に近い項目だけ確認する。

例:

- `calendar.tsx` のドット表示
- 日付タップの詳細
- 下余白

この時のルール:

- 全体スキャンしない
- 10〜15項目に絞る
- build はしない

### レベル3: 提出前フルQA

TestFlight 候補ができたら最後に全体確認する。

この時だけ:

- 認証
- 未接続導線
- 約束
- カレンダー
- ごほうび
- 設定
- 通知
- セキュリティ

をまとめて見る

## タブごとに進めるのは有効か

はい、とても有効です。

おすすめ順:

1. 認証 / オンボーディング
2. 未接続導線
3. ホーム
4. 約束一覧
5. 約束作成 / 編集
6. カレンダー
7. ごほうび
8. 設定
9. API / セキュリティ

理由:

- 触るファイルが小さくまとまる
- QA項目を絞りやすい
- どの不具合がどの変更で起きたか追いやすい

## 毎ターンの指示テンプレ

### Codex 向け

以下の形で狭く依頼する:

```text
このターンでは create-rule.tsx と edit-rule.tsx だけ触ってください。
やることは3つだけです。
1. いつから始めるかを消す
2. 何時ごろの約束かを消す
3. リマインドの見た目崩れを直す
他の画面には広げないでください。
```

### Claude 向け

```text
まず handoff ファイルを読んでください。
今回は calendar.tsx だけを scoped QAしてください。
確認したいのは
1. ドット表示
2. 日付タップ時の詳細
3. 下余白
の3つだけです。
build はまだしないでください。
```

## 毎ターンの上限目安

- 対象テーマ: 1つ
- 対象ファイル: 1〜3本
- スクショ: 1〜3枚
- 確認項目: 3〜10個
- ログ貼り付け: 必要部分だけ

## QAの分け方

### 毎回やる最小スモーク

- 起動する
- クラッシュしない
- 今回触った画面が開く
- 保存 / 遷移の主要導線が死んでいない

### 差分QA

- 今回の修正対象だけ

### 提出前QA

- 全項目

## handoff 運用

長い会話履歴を正本にしない。
毎回、意味のある区切りごとに handoff を更新する。

最低限残す内容:

- 今回直したこと
- まだ残っていること
- 次に触るファイル
- テスト結果
- Claude に渡す時の注意

## やってはいけないこと

- 毎回フルQA
- 毎回全画面スキャン
- 毎回大きいチェックリスト全文をチャットに貼る
- 1ターンで複数タブを同時に直す
- QAと新機能追加を同じターンでやる

## 実務的なおすすめ

Pairlog は今後この形で進めるのが最も安定します。

1. タブ単位で修正
2. handoff 更新
3. Claude に scoped QA
4. 問題なければ次タブへ
5. 最後だけ pre-build QA

## 今の結論

コンテキスト圧縮の最大要因は、

- 広すぎる依頼
- 長い履歴
- 大量の貼り付け

です。

したがって、最も効く対策は

- タブごと
- 画面ごと
- バグごと

に小分けして進めることです。
